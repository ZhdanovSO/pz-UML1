## 1. Діаграма варіантів використання (Use Case Diagram)

```mermaid
graph LR
    subgraph "Користувачі"
        User(("Співробітник"))
        Admin(("IT-Адмін"))
    end

    subgraph "Система замовлення техніки"
        UC1("Переглянути каталог")
        UC2("Створити заявку")
        UC3("Відстежити статус")
        UC4("Керування складом")
        UC5("Затвердити/Відхилити заявку")
    end

    User --- UC1
    User --- UC2
    User --- UC3

    Admin --- UC4
    Admin --- UC5
    Admin --- UC3
```

## 2. Діаграма послідовності (Sequence Diagram)

```mermaid
sequenceDiagram
    participant Співробітник
    participant Система
    participant Адміністратор

    Співробітник->>Система: Запит каталогу
    Система-->>Співробітник: Список техніки
    Співробітник->>Система: Надсилання заявки
    Система->>Адміністратор: Нова заявка на розгляд
    
    Note over Адміністратор: Перевірка наявності
    
    alt Схвалено
        Адміністратор->>Система: Підтвердити
        Система-->>Співробітник: Статус: Схвалено
    else Відхилено
        Адміністратор->>Система: Відхилити
        Система-->>Співробітник: Статус: Відхилено
    end
```

## 3. Діаграма діяльності (Activity Diagram)

```mermaid
stateDiagram-v2
    [*] --> Авторизація
    Авторизація --> ПереглядКаталогу
    ПереглядКаталогу --> ЗаповненняЗаявки
    
    ЗаповненняЗаявки --> ПеревіркаДаних
    
    state ПеревіркаДаних <<choice>>
    ПеревіркаДаних --> ЗаповненняЗаявки : Помилка
    ПеревіркаДаних --> РозглядАдміном : Успішно
    
    state "Рішення Адміна" as Decision <<choice>>
    РозглядАдміном --> Decision
    
    Decision --> Схвалення : ТАК
    Decision --> Відмова : НІ
    
    Схвалення --> ВидачаТехніки
    Відмова --> Архів
    
    ВидачаТехніки --> Архів
    Архів --> [*]
```
У цій практичній роботі було спроєктовано систему замовлення IT-обладнання. За допомогою UML-діаграм продемонстровано:

1. **Діаграма варіантів використання (Use Case):** Визначає основні ролі (Співробітник, IT-Адмін) та їхні права в системі — від простого перегляду каталогу до управління складом та затвердження заявок.
2. **Діаграма послідовності (Sequence):** Показує покрокову взаємодію між користувачем, системою та адміністратором у часі. Деталізовано процес перевірки наявності техніки та зміну статусів заявки залежно від рішення.
3. **Діаграма діяльності (Activity):** Відображає логічний алгоритм обробки запиту: перевірку коректності введених даних, розвилки рішень адміністратора (схвалення або відхилення) та фінальні етапи (видача техніки з подальшим архівуванням).
